LLM Council Report — Oskar Bulk Item Upload

Scanfil APAC · ECN Platform · 17 June 2026
Question: Is the proposed bulk CSV/Excel import architecture (react-dropzone + SheetJS client-side → preview → /bulk endpoint) the best approach for Oskar? What failure modes are being missed, and is the UX pattern optimal?
The One Thing To Do First

Write the header-fingerprint validator before writing a single line of parsing logic.

Define the canonical column header list as a constant. On every upload — before parsing, before preview, before any network call — check that the uploaded file's headers match that list. If they don't, block immediately with a message naming the missing or unexpected columns. This costs two hours to build and future-proofs every downstream assumption your parser makes. Without it, template drift silently corrupts data. With it, everything else can be built with confidence.

Council Verdict

Chairman Synthesis

Where the Council Agrees

  • Dedicated /bulk endpoint: unanimous. No sequential loop, no background job.
  • All-or-nothing transaction: unanimous. Stop entirely on any error — the user corrects and retries.
  • Two-step preview/confirm UX is correct in principle. The preview is the primary trust mechanism — do not remove it to simplify the POC.
  • SheetJS must use {raw: false} — all columns read as strings, never coerced to numbers. AS/400 item numbers frequently carry leading zeros (e.g. 0034567 → silently becomes 34567). This is a data integrity hole, not a theoretical edge case. Non-negotiable.
  • DRAFT status race condition is a ship-blocking bug: ECN status must be checked inside the transaction with a row-level lock, not in application code before the transaction starts.
  • Skip rows by header content match, not position. Hardcoded "skip rows 2-6" is a comment waiting to become a bug.

Where the Council Clashes

Client-side vs server-side parsing — the central dispute.

Executor (server-side): openpyxl + stdlib csv, zero new JS dependencies, one place to fix template changes. Simpler to debug.

Contrarian + First Principles (client-side): The preview is the value proposition. Engineers need to see their data reflected back instantly — before any network round-trip. Eliminating the preview to simplify the POC trades away the primary reason engineers will trust bulk upload over manual entry.

Chairman resolution: Do both. Parse client-side with SheetJS ({raw: false}) for instant preview. Also POST the raw file to the backend as multipart/form-data. The backend runs an authoritative dry-run Pydantic validation pass — returns structured errors with row numbers if anything fails. If dry-run passes, re-validate and INSERT inside a single transaction. Client-side validation is for UX speed only. The backend is the source of truth. Neither trusts the other.

Idempotent upsert vs pure insert: The Expansionist argues for idempotent upsert keyed on item_number for retry safety. For a POC with atomic batches, a pure insert with a DB-level uniqueness constraint on (ecn_id, item_number) is correct and simpler. Upsert is the right shape for Sprint 2 production hardening.

Blind Spots the Council Caught

1. Authorization missing from the architecture. All 5 peer reviewers flagged this independently. A /bulk endpoint that bypasses the role gates applied to single-item add is a compliance gap. Scanfil APAC is a regulated electronics contract manufacturer under ISO 13485. The bulk endpoint must enforce identical RBAC checks: who is permitted to add items to this specific ECN. Wire this in on day one — if the POC ships without it, it becomes the production behavior by inertia.

2. No server-side file guards. Do not trust browser MIME detection. Enforce <1MB and whitelist .xlsx/.csv content-type at the endpoint level before any parsing begins. Define behavior on VPN timeout mid-upload — the transaction either committed or it didn't; the client needs a clear error state, not an orphaned spinner.

3. Encoding. CP1252 from AS/400/MOVEX exports is a real vector. Show raw cell values in the preview, not normalized ones. Use chardet on the server side if accepting raw .csv files.

The Full Recommendation

  • Client-side parse: SheetJS with {raw: false}. Header-fingerprint check first — block on mismatch before preview renders.
  • Preview: Full grid with inline error highlighting. Show raw cell values. This is not optional for the POC.
  • Confirm → Backend: POST raw file as multipart/form-data. Backend dry-run Pydantic pass first. If clean, INSERT inside a single transaction. ECN DRAFT status checked via SELECT ... FOR UPDATE inside the transaction — not in application code outside it.
  • DB: Unique constraint on (ecn_id, item_number). Line numbers auto-assigned by sequence.
  • Auth: Identical RBAC as single-item add on the /bulk endpoint. Day one, non-negotiable.
  • File guards: Server-side content-type whitelist + size enforcement before parsing begins.
  • Skip logic: Header name matching, not positional row numbers.
  • SheetJS {raw: false}: Non-negotiable. Leading-zero integrity for MOVEX item numbers.

Advisor Responses (full)
The Contrarian Risk Focus

The fatal flaw is the preview step becoming a trust trap. Engineers will glance at a 60-row preview table, see no red highlights, click Confirm, and then discover the backend rejected everything because of a Pydantic constraint the client-side validator didn't cover. Field length limits drift. Enum values get added server-side and forgotten on the client. That gap is where engineers lose 20 minutes at 4pm on a Friday.


The specific failure mode nobody asked about: Excel files from MOVEX exports carry numeric formatting on "text" fields. SheetJS will silently coerce a part number like 0034567 to the integer 34567, stripping the leading zero. Your Pydantic model accepts it. Your MOVEX integration rejects it at sync time — three steps later, in a different session, with no connection back to the upload. That's a data integrity hole.


Stop-on-error all-or-nothing is correct — but apply it after a dry-run server validation pass, not after a client-side pass. Force SheetJS to read all columns as raw strings: {raw: false}. Non-negotiable.

The First Principles Thinker First Principles

What are you actually solving? Engineers have a standardised Excel template. They want its rows to become ECN items. That's it. This is not a general file upload problem. It's a known-schema, trusted-user, small-payload data entry shortcut.


Client-side parsing is correct — not because SheetJS is great, but because you're reading a template you control. Parse it where the feedback loop is tightest: in the browser, before any network round-trip. The preview step isn't a UX nicety, it's the core value proposition — the moment engineers see their data reflected back before commit is the moment they stop fearing bulk operations.


The failure mode nobody named: The template itself will drift. Engineers will add columns, rename headers, use merged cells. The missing piece is a header-fingerprint check on upload that detects template version mismatch and blocks with a clear message before the preview even renders. Ship that first.

The Expansionist Upside Focus

The standardised Excel template is the real unlock — not the upload mechanism. The same parser can be extended to ingest from M3 MOVEX BOMs directly via DB2 direct endpoints (MITMAS and MITPOP are already queryable). The template becomes optional. Cross-ECN item reuse and change velocity analytics are hiding in these CSV rows.


Client-side SheetJS is correct for the POC because the preview-before-confirm UX is worth more than any backend-side flexibility gain. The two-step flow is the trust mechanism that makes engineers adopt it over manual entry.


The dedicated /bulk endpoint is right, but shape it as an idempotent upsert keyed on item_number within the ECN, not a pure insert. "Upload failed at row 47, fix and retry" becomes a non-event — the retry just overwrites what was already written cleanly. Retry safety is free if you design for it now. (Chairman note: deferred to Sprint 2 — pure insert + DB uniqueness constraint is correct for the POC.)

The Outsider Fresh Eyes

The template skip logic is a hidden landmine. Rows 2-6 are skipped as "instruction rows." What happens when an engineer deletes the instructions because they find them annoying? Or when the template gets updated and the instruction block grows to row 8? Skip rows by content match, not position. Match column headers by name. Everything that doesn't match a known header pattern gets ignored.


The DRAFT status race condition. Engineers will upload, see a preview, confirm — and then get a backend error because the ECN slipped out of DRAFT status between their preview and their confirm click. Someone else approved it in that window. You have no optimistic lock. The confirm button needs to check ECN status server-side before inserting, and the error message must say exactly why it failed.


Duplicate detection is unspecified. Upload the same file twice → 160 items instead of 80? Item number uniqueness must be enforced at the database constraint level, not just application logic.


Encoding. MOVEX data on AS/400 often comes out in CP1252. Show the raw cell value in the preview, not a normalized version.

The Executor Ship It

Parse server-side. Send the raw file as multipart/form-data. Use openpyxl + stdlib csv. Zero new JS dependencies. If the template format ever changes, fix it in one place. (Chairman note: resolved — parse both sides; client for preview UX, server for authoritative validation.)


Dedicated /bulk endpoint. The sequential loop is a trap — 80 round-trips, 80 failure points, no atomicity. One endpoint, one transaction, one commit or rollback.


Two-step UX is correct but simplify it — skip the in-browser preview grid for the POC. Show errors as a numbered list. (Chairman note: preview grid is not optional — it's the trust mechanism. Don't remove it.)


Failure modes: duplicate item_number in batch, CP1252 encoding (use chardet), stale ECN status check inside the transaction not before it.
