Blind Spots the Council Caught

Critical: Does routing data already exist in Movex? If Branko's template is sourced from an existing Movex routing record, OSKAR does not need to own this write path at all. Nobody asked. This is the most important unasked question in the session.

If no, the 30-minute target may be achievable without routing in Sprint 2.

Approval gates are missing from the design. The ECN workflow has 13 statuses and 16 roles. A routing write to Movex is an engineering-controlled document change. It must pass through the approval chain before hitting production ERP. No advisor modelled this. Fire-and-forget outbox for routing is an audit and compliance gap.

The outbox schema has an integrity problem. If no MI program is confirmed, outbox_type for routing entries has no valid value. The schema must include a typed status = 'pending_mi_confirmation' sentinel — not a nullable FK — to avoid silent schema rot.

The Recommendation

Add the routing table in Sprint 2. Do not add the outbox entry type. Do not build the Excel importer.

Before writing any migration, answer two questions this week — both are half-day investigations:

  • Does routing already exist in Movex? Ask Branko directly. If yes, OSKAR's routing table becomes audit capture only, and the outbox write discussion is moot for Phase 1.
  • Is a routing write required for ECN closure? If routing is a downstream post-approval action via a separate Movex user step, the outbox write has no go-live dependency. The table captures data; integration is Sprint 3 or later.

If both answers confirm OSKAR must own this data and write it at ECN closure:

  • Add ecn_routing_operations with a typed status field including pending_mi_confirmation — no nullable outbox FKs.
  • Run the MI spike in parallel this sprint, against CONO=300.
  • Wire the outbox entry type only when MI program is confirmed and tested.
  • Do not commit the outbox write as a Sprint 2 deliverable. It is a Sprint 2 candidate, conditional on the spike.
  • Excel import is Sprint 3. Karen gets informed, not ignored.

The approval gate question must also be answered before outbox write is designed. Routing writes cannot be modelled as background fire-and-forget through a 16-role approval workflow.
