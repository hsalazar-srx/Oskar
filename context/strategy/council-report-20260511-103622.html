Chairman's Verdict

Agrees Where the Council Agrees

The stack is not the question. Every advisor converged: Vite + React + TypeScript is settled by SM-Portal's existence. The live system next door made the decision. No advisor recommended a divergent framework, and on-premise static deployment sealed it — Next.js and Remix add SSR complexity with zero benefit on IIS or a Linux VM without a Node runtime.

SSE is not a stack decision. It is an afternoon of work with useEffect and EventSource.

The council also agrees: raw Tailwind alone is a trap for a solo backend-leaning developer. A component library is mandatory, not optional.

Clashes Where the Council Clashes

shadcn/ui vs. Ant Design. The real fault line.

For Ant Design (Contrarian + Outsider): Table virtualization, multi-step forms, modal management, and approval workflow components ship out of the box. For an 8-week runway with one backend-leaning engineer, ready-made enterprise component density reduces risk against the June deadline. You don't have design time — Ant Design doesn't ask for it.

For shadcn/ui (Expansionist + Executor): You own the source. No library API surface to fight when approval workflow logic needs non-standard behavior. Tailwind coherence with SM-Portal is structural, not cosmetic. Every component built for Oskar — the BOM line editor, the approval timeline, the outbox error panel — becomes a reusable asset for WMS and PLM.

Separate app vs. SM-Portal module. First Principles argued Oskar should be a route namespace inside SM-Portal. Every peer review rejected this: Oskar deploys to a separate Linux VM, has a different release cadence, a different stakeholder group, and a Python/FastAPI backend rather than .NET. This question was closed by deployment reality.

Blind Spots What the Council Missed

1. Authentication integration cost. SM-Portal runs .NET JWT on SRXWEBAPP1. Oskar runs FastAPI issuing its own JWTs on a separate Linux VM. Two separate origins, on-premise, no API gateway. For an approval workflow with role-based action buttons — where the wrong token claim means the wrong user approves a BOM change — SSO or token-sharing strategy is a day-one architectural decision, not a polish item. Every advisor evaluated component libraries while ignoring the auth surface entirely.

2. OpenAPI-to-TypeScript client generation. FastAPI auto-generates an OpenAPI schema. Tools like orval or openapi-typescript consume that schema and produce type-safe API clients, TanStack Query hooks, and Zod schemas automatically. For a solo backend-leaning developer, this eliminates an entire category of frontend bugs and reduces ongoing maintenance burden more than any component library choice. This is the highest-leverage tooling decision available — and it went unmentioned by all five advisors.

Recommendation The Recommendation

Confirm the stack: Vite + React + TypeScript + Tailwind CSS.

Choose shadcn/ui, not Ant Design. The June deadline is real, but it does not override the platform argument. Ant Design's density is appealing until the approval workflow needs behavior the library doesn't expect — and enterprise approval workflows always do. At that point, the time saved on initial scaffolding is consumed fighting the library's opinions. shadcn/ui components live in your repo; you fight nothing.

The mandatory additions — non-optional:

  • React Hook Form + Zod — nested BOM forms and MPN grids
  • TanStack Query — SSE integration and optimistic updates on the approval panel
  • Zustand or XState — approval workflow state machine. The Contrarian's warning here is the most important technical caution in the entire council. The state machine complexity will make or break month three regardless of component library.

On auth: Before writing a single React component, define the JWT strategy. Options: shared signing key between FastAPI and SM-Portal's .NET backend (simple, fragile), a lightweight on-premise identity service (correct, more work), or accept separate sessions with explicit login per app (acceptable for v1, document the tradeoff). Pick one, write it down, build the API client layer against it.

Next The One Thing to Do First

Set up orval against FastAPI's /openapi.json before writing any component code. Configure it to generate TanStack Query hooks with Zod validation and commit the generated client to the repo. From that point forward, every API response the backend emits is automatically type-safe on the frontend, every breaking API change surfaces as a TypeScript compile error, and the solo backend-leaning developer can work in Python with confidence that the TypeScript layer stays synchronized. This single step — 2 hours to configure correctly — removes more frontend risk than any component library decision on the board.

Full Advisor Responses
Contrarian State architecture is the real decision

The "visual coherence with SM-Portal" constraint is doing a lot of heavy lifting here, and nobody is questioning it hard enough. SM-Portal is already in production. That's a sunk cost argument dressed up as a technical requirement. The real question is: how much does cross-app visual coherence actually matter when engineers are the users? They navigate between tools constantly. They don't care if the table borders match.

Now for the actual flaw in the proposal: Tailwind without a component library is a trap for a solo backend-leaning dev. Raw Tailwind for complex approval workflows, nested BOM form grids, and multi-state filter panels will consume weeks of a backend dev's time reinventing wheels. You will build a bad Ant Design from scratch.

The honest answer: if you're picking Vite + React + TypeScript, you must commit to Ant Design or shadcn/ui on day one, not optionally. Ant Design wins for this use case — it ships table virtualization, form validation, multi-step workflows, and modal management out of the box. A backend-leaning dev can be productive in days.

The deeper risk nobody is naming: the approval workflow state machine. React alone doesn't save you here. Without a proper state management approach (Zustand, TanStack Query, or XState), the approval screen will become unmaintainable by month three regardless of which CSS framework you pick. Stack choice is the wrong obsession. State architecture is the real decision.

First Principles Why does Oskar need a custom frontend at all?

The framing assumes the decision is "which frontend stack." It isn't. The real question is: why does Oskar need a custom frontend at all?

SM-Portal already exists, is in production, and is built on exactly the stack being evaluated. Oskar's screens — filtered lists, complex forms, approval workflows, SSE updates — are not novel UI problems. They are solved problems within SM-Portal's existing architecture. The team is one backend-leaning engineer with a finite runway.

The first-principles answer: Oskar should be a module inside SM-Portal, not a separate application. Two separate frontends means two build pipelines, two deployment targets, two authentication integration points, two places to maintain shared components, and doubled cognitive overhead for a solo developer. That is not a tradeoff — it is pure waste.

The stack question only becomes real if there is a firm organizational reason Oskar must be a separate application — political boundary, different deployment server, separate release cadence. If any of those exist, state them explicitly. If none exist, the council is being asked to evaluate paint colors for a house that should share a wall with the one already built.

Note: peer reviews unanimously rejected this position — deployment reality (separate Linux VM, separate FastAPI backend, separate release cadence) closes this question before it can be strategic.

Expansionist Oskar is Scanfil APAC's frontend platform seed

The question is wrong. The real question is: what stack makes Oskar extensible enough to become the foundation for Scanfil APAC's entire modernisation programme?

Confirm the React + Vite + Tailwind stack. That's not the interesting decision. The interesting decision is shadcn/ui over Ant Design, and the reason isn't aesthetics — it's ownership. shadcn/ui gives you the component source code. Every table, every modal, every form field lives in your repo. When Oskar evolves from ECN workflow into PLM integration, WMS handoffs, and eventually supplier-facing portals, you're not negotiating with an opinionated library's API surface. You own the primitives.

The upside everyone's undervaluing: Oskar isn't just an ECN tool. It's the React component foundation for every future Scanfil APAC internal app. The BOM line editor, the approval timeline component, the outbox error panel — these are reusable enterprise building blocks. Build them as clean, composable shadcn/ui components from day one and you're simultaneously building the design system for WMS, PLM, and whatever comes after IFS migration.

TanStack Query deserves a specific callout for SSE + optimistic updates. Combined with Zod + React Hook Form for the complex nested BOM forms, the "solo backend-leaning dev" problem largely disappears. Skip Next.js — Vite + static build is a single dist/ folder. Deploy anywhere. No runtime to maintain.

Outsider You're choosing what matches the portal, not the problem

The framing of this question already reveals the answer you're going to choose, which is the problem. "Visual coherence with SM-Portal is required" is doing enormous hidden work here. That's not a technical constraint — it's a social one. Challenge that assumption before it locks you in.

What an outsider sees immediately: you have one backend-focused engineer building complex approval workflows, nested BOM forms, and real-time panels. The bottleneck is not the framework — it's component density. Ant Design with plain React solves this faster than any combination involving "assemble your own component system with shadcn/ui and Tailwind."

shadcn/ui is an excellent choice when you have design time. You don't. The curse of knowledge here is that "Tailwind is simple" feels true to people who already know it. To a backend-leaning developer building a multi-step approval form with dynamic BOM line items, Tailwind means writing layout CSS from scratch, just with different names. Ant Design's Table, Form, Modal, Steps components are immediately usable with zero styling decisions.

The on-premise deployment constraint eliminates the actual advantages of Next.js and Remix. You'd be adopting their complexity with none of their benefits. Ant Design + React + TypeScript + Vite gives you 80% of your screens out of the box. The "visual coherence" concern can be addressed with a shared color palette — not a shared component system.

Executor Use what SM-Portal uses. The only question is which library.

Use what SM-Portal already uses. Full stop. The stack is already decided by the production system sitting next to this one. The only real question is which component library to drop on top.

Monday morning action: Install shadcn/ui. Run npx shadcn-ui@latest init. Done in 20 minutes.

Why shadcn wins: shadcn components are copy-paste into your codebase — no version conflicts, no fighting the library's opinions when your approval workflow needs non-standard behaviour. Tailwind is already there, so styling coherence with SM-Portal is automatic, not a negotiation. React Hook Form + Zod integrations are documented.

What you don't do: Not Next.js (no SSR needed, IIS complexity, zero benefit). Not Remix (same problem, unfamiliar). Not SvelteKit (forces a parallel ecosystem, doubles maintenance).

SSE in React is three lines. useEffect + EventSource. No library needed. The actual bottleneck is the approval workflow state machine and nested form validation — not the framework. React Hook Form with useFieldArray handles BOM lines directly.

First step after shadcn init: scaffold the ECN list page using shadcn DataTable with TanStack Table. That's your proof-of-concept, built in a day.
